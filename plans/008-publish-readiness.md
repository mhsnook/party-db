## Why this matters

The package is being prepped for its first npm publish (commit `3779114`). Three
gaps: (1) every peerDependency is `"*"`, so consumers resolve untested majors —
the repo's own docs say behavior was "Verified on @tanstack/db@0.6.10", a 0.x line
where minors break; (2) the manifest lacks `sideEffects` (blocks tree-shaking) and
`engines` (Node 22 is required — `node:sqlite` for the bench, and CI pins 22);
(3) there is no release workflow — a first publish would be a manual, unattested
`pnpm publish` with no CI gate on the artifact and no provenance.

## Steps

### Step 1: Pin the peers and mirror them in devDependencies

Re-verify installed versions
(`node -pe "require('./node_modules/@tanstack/db/package.json').version"` etc.),
then set:

```json
"peerDependencies": {
  "@standard-schema/spec": "^1.1.0",
  "@tanstack/db": "^0.6.10",
  "partyserver": "^0.5.8",
  "partysocket": "^1.3.0"
}
```

(`^0.x.y` ranges stay within the 0.x minor — exactly the "verified on" guarantee.)
Keep `peerDependenciesMeta` as-is. Add the same four packages at the same ranges
to `devDependencies` so a fresh clone resolves deterministically, then
`pnpm install` to update the lockfile.

**Verify**: `pnpm install` exit 0; `git diff pnpm-lock.yaml` shows only the four
packages moving into the importer's devDependencies (no version changes);
`pnpm typecheck && pnpm test && pnpm test:integration` → green.

### Step 2: Manifest hygiene

Add to `package.json`:

```json
"sideEffects": false,
"engines": { "node": ">=22" }
```

(`sideEffects: false` is safe: every module is declarations + exports; the only
top-level statements are class/function/type definitions — spot-check
`src/client/party-db.ts` and `src/server/index.ts` for stray side effects before
asserting it.)

**Verify**: `pnpm build && pnpm publish --dry-run --no-git-checks` → tarball lists
`dist/**`, `src/**`, `README.md`, `LICENSE`, `package.json` and nothing else
(no `plans/`, no `test/`, no `docs/`).

### Step 3: CI pack check

In `ci.yml`, after the integration test step, add:

```yaml
      - run: pnpm publish --dry-run --no-git-checks
```

**Verify**: `git diff .github/workflows/ci.yml` shows only that addition. (CI
executes on push; locally the same command already passed in Step 2.)

### Step 4: Release workflow

Create `.github/workflows/release.yml`: triggered on tags matching `v*`; runs the
same setup as `ci.yml` (checkout, pnpm/action-setup, setup-node 22 + pnpm cache,
frozen install); then typecheck + build + both test suites; then publish with
provenance. Skeleton:

```yaml
name: Release
on:
  push:
    tags: ['v*']
permissions:
  contents: read
  id-token: write # npm provenance
jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
          registry-url: https://registry.npmjs.org
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm test
      - run: pnpm test:integration
      - run: pnpm publish --provenance --access public --no-git-checks
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Check pnpm 10's documented provenance flag first (`pnpm help publish` or the pnpm
docs); if `--provenance` is not supported, publish via
`npm publish --provenance --access public` after `pnpm build` (npm reads the same
`prepack`? — no: run `pnpm build` explicitly first in that variant). Note in the
workflow file (comment) that `NPM_TOKEN` must be configured as a repo secret
before the first tag — that's an operator action, not yours.

**Verify**: `pnpm exec node -e "require('js-yaml')"` is not available — instead
validate YAML by running `pnpm dlx yaml-lint .github/workflows/release.yml` OR
simply `node --experimental-strip-types -e "…"`-free check: use
`python3 -c "import yaml,sys; yaml.safe_load(open('.github/workflows/release.yml'))"`
if python3 exists, else visual inspection + `git diff`. (Any one of these passing
is sufficient.)
